# Solana Data Model — Quasar Reference

Read this before building any new program. It covers the mental model that Quasar programs operate within.

---

## Accounts

Everything on Solana is an account. An account has:

| Field | Type | Notes |
|---|---|---|
| `address` | `[u8; 32]` | The unique key (pubkey) |
| `lamports` | `u64` | Balance in lamports (1 SOL = 1,000,000,000) |
| `data` | `Vec<u8>` | Arbitrary byte storage |
| `owner` | `[u8; 32]` | Which program owns this account |
| `executable` | `bool` | True only for deployed programs |

A program can only **write** to accounts it owns. It can read any account. It can debit lamports from accounts that signed the transaction.

---

## Account Ownership

- System program owns all wallets and uninitialized accounts
- Your program owns accounts it creates via `init` or `init_if_needed`
- Only the owning program can modify `data` or `lamports`
- Closing an account = zeroing data + transferring lamports out + reassigning owner back to system

---

## PDAs (Program Derived Addresses)

PDAs are addresses derived from seeds + program ID. They have no private key, so only your program can sign for them via `invoke_signed`.

```
PDA = sha256(seed1 ++ seed2 ++ ... ++ program_id ++ "ProgramDerivedAddress") % curve_order
```

In Quasar:
```rust
// Derive and verify in accounts struct:
#[account(seeds = [b"vault", signer], bump)]
pub vault: &'info mut UncheckedAccount,

// Use stored bump for CPI signing (cheaper than re-deriving):
#[account(seeds = [b"vault", signer], bump = vault.bump)]
pub vault: &'info mut Account<Vault>,

// Access bump in handler:
let seeds = bumps.vault_seeds();   // generated method
```

Off-chain PDA derivation (in tests):
```rust
let (vault, bump) = Address::find_program_address(
    &[b"vault", user.as_ref()],
    &crate::ID
);
```

Seeds can be: byte literals, `signer` (expands to signer.address()), field references, byte slices.

---

## Rent

Solana charges rent for storing data. Accounts must maintain a minimum balance (rent-exemption) or be closed. In practice, always fund accounts to rent-exempt level.

Minimum balance ≈ `(data_bytes + 128) * lamports_per_byte * 2.0`

In Quasar, `init` and `init_if_needed` handle rent automatically when you set `payer` and `space`.

Accessing rent on-chain:
```rust
// In accounts struct:
pub rent: &'info Sysvar<Rent>,

// Compute minimum balance:
let min = self.rent.minimum_balance(data_size);
```

## Instructions and Transactions

A transaction contains one or more instructions. Each instruction specifies:
1. Which program to call
2. Which accounts to pass
3. Instruction data (discriminator + args)

Instructions execute sequentially in a transaction. If any fails, all changes roll back.

In Quasar, the discriminator is the first byte(s) of instruction data. `#[instruction(discriminator = N)]` sets this.

---

## Cross-Program Invocation (CPI)

A program can call another program during instruction execution.

**Normal CPI** — when the caller's account signs:
```rust
self.system_program.transfer(self.payer, self.dest, amount).invoke()?;
```

**Signed CPI** — when a PDA signs (only the owning program can do this):
```rust
let seeds = bumps.vault_seeds();   // &[&[u8]] seed slices including bump
self.system_program.transfer(self.vault, self.dest, amount).invoke_signed(&seeds)?;
```

The `bumps.field_seeds()` methods are generated by `#[program]` for every PDA in the accounts struct.

---

## Account Lifecycle in a Quasar Program

```
1. INIT       — create account, set size, fund to rent-exempt, set owner to your program
2. USE        — read/write data via zero-copy accessor, CPI as needed
3. REALLOC    — resize account data (for dynamic fields — Quasar handles this automatically)
4. CLOSE      — transfer lamports out, zero data, reassign owner to system
```

In constraints:
- `init` → creates and funds the account
- `init_if_needed` → creates only if not already initialized (check discriminator)
- `close = dest` → closes and sends rent to `dest`
- `space = T::SIZE` → byte size (use the generated constant)

---

## Space Calculation

For fixed accounts, use the generated `T::SIZE` constant:
```rust
#[account(init, payer = signer, space = Vault::SIZE)]
pub vault: &'info mut Account<Vault>,
```

For accounts with dynamic fields (`String`, `Vec`), Quasar computes and reallocates space automatically when you call `set_inner` or `set_<field>`.

---

## Signers

An account is a signer if its keypair signed the transaction. Programs verify this via:
```rust
pub signer: &'info mut Signer,   // constraint: must be signer
```

Or manually on a remaining account:
```rust
if !account.is_signer() {
    return Err(ProgramError::MissingRequiredSignature);
}
```

PDAs never sign transactions — they "sign" CPI via `invoke_signed` + bump seeds.
